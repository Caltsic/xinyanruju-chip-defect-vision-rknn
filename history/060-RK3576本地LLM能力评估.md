## Public Hardware/Software Facts

- RK3576 has a quad Cortex-A72 + quad Cortex-A53 CPU and a 6 TOPS INT8 NPU.
- RK3576 NPU supports int4/int8/int16/FP16/BF16/TF32 per Rockchip brief datasheet.
- RKLLM/RKNN-LLM ecosystem documentation and third-party integration notes list RK3576 support, but it still needs board-specific smoke testing against the installed kernel, NPU driver, and runtime.

## Engineering Assessment

The board can run a lightweight local LLM, but not as a comfortable always-on peer to the current realtime vision stack.

Practical limits on this 4GB board:

```text
Comfortable alongside vision: 0.5B class, 4-bit quantized
Practical upper bound alongside vision: 1.5B class, 4-bit quantized, short context, low priority
Possible when vision is stopped or reduced: 3B class, 4-bit quantized, short context
Not recommended on this board: 7B class, because model + KV cache + OS + vision memory exceed the practical 4GB envelope
```

The current vision chain already occupies the NPU in two-stage RKNN inference and delivers about 6.3-6.7 FPS. A NPU-backed LLM would contend with chip OBB and defect segmentation inference. If LLM must run while realtime detection continues, CPU-only llama.cpp-style inference is safer operationally, but it will be slow and should be constrained with low thread count, short context, and low process priority.

## Recommended Next Step

For a board-only proof of concept:

1. Start with CPU `llama.cpp` and a small GGUF model:
   - `Qwen2.5-0.5B-Instruct` or similar, Q4 quantization.
   - Then try `Qwen2.5-1.5B-Instruct` Q4.
2. Use context length 1024 or 2048.
3. Run with low priority and limited threads, then measure vision FPS drop.
4. Only after that test RKLLM/NPU, and only as an on-demand mode or when vision is paused.

Do not store LLM models under `/userdata`; only about 545MB was free there. Use the root filesystem path with about 11GB free, or external storage.
