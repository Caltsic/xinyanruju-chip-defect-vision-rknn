# 硬件视觉链路开发注意事项

更新时间：2026-05-04

## 快速结论

后续调试优先按这个顺序排查：

```text
补光/白平衡 -> UVC 模式 -> 采集格式 -> 单路占用 -> ROI 尺度 -> 预处理 -> 模型类别/标签
```

当前最容易误判的点是：相机画面“人眼清楚”不等于模型能直接全图检测。现有缺陷模型没有 `chip` 类，且当前芯片在全幅里偏小；应先做芯片 ROI，再跑缺陷检测。

## WS2812 补光与色偏

- WS2812-8 环形灯接线：

```text
WS2812 VCC / 5V  -> 泰山派 40Pin 物理 2 或 4 脚 5V
WS2812 GND       -> 泰山派 40Pin 物理 20 脚 GND
WS2812 DI / DIN  -> 泰山派 40Pin 物理 19 脚 SPI1_MOSI(M1)
WS2812 DO / DOUT -> 不接
```

- 当前板端需要 `/dev/spidev1.0`。换板或系统重刷后如果没有该节点，先执行：

```powershell
F:\anaconda\python.exe .\tools\adb_ws2812_ring.py status
F:\anaconda\python.exe .\tools\adb_ws2812_ring.py deploy --reboot
```

- 全白 `255,255,255` 会让当前相机画面明显偏紫。实测背景均值从 `RGB 134/124/193` 调到 `166/162/163` 的推荐命令是：

```powershell
F:\anaconda\python.exe .\tools\adb_ws2812_ring.py set --rgb 190,255,100 --brightness 0.5 --count 8
```

- 上面命令经过亮度缩放后，板端实际输出约为：

```text
rgb=(95, 128, 50)
```

- 若长期从板子 5V 排针取电，不建议 8 颗灯珠长时间高亮全白；全白满亮约 `480mA`。高亮测试建议外接 5V，并与泰山派共地。

## MaixCAM Pro UVC 注意点

- MaixCAM Pro 默认不是普通 UVC 摄像头，刚接入时可能只枚举 RNDIS/NCM 网络接口。
- 必须在 MaixCAM 屏幕打开官方 `UVC Camera` 应用，并看到：

```text
UVC started. Please use 'Guvcview'
and mjpeg channel.
```

- 只有官方 `UVC Camera` 应用接管后，`MJPG 1280x720` 才是真实相机画面。
- 仅看到 `/dev/video73` 不代表是真实画面；默认 feeder 可能输出 `/bin/cat_224.jpg` 静态测试图。
- 不要单独执行：

```bash
/etc/init.d/uvc_tool.sh stop_server
```

该操作实测会让 MaixCAM USB gadget 掉线，常需要物理复位/断电重插恢复。

- 当前 MaixCAM UVC 格式结论：

```text
MJPG 1280x720  可用，真实相机画面
MJPG 640x480   可枚举，但应用可能仍给 1280x720 JPEG
YUYV 640x480   实测全黑，不要用于检测
```

## MJPEG 坏帧与设备占用

- MaixCAM 的 MJPEG UVC 流偶发坏包，不是每个 V4L2 buffer 都是完整 JPEG。常见日志：

```text
premature end of data segment
does not contain JPEG SOI marker
Not a JPEG file: starts with ...
```

- 处理策略：单个坏 MJPEG 帧必须跳过并 requeue，不能让进程退出。当前 `rknn_chip_defect_maixcam_stream` 已实现：
  - 在 buffer 内搜索 JPEG `FFD8` SOI 起始标记。
  - 单帧坏包跳过。
  - 连续 60 个坏包才退出。

- `/dev/video73` 不能被多个采集进程同时打开。若出现：

```text
VIDIOC_S_FMT 1280x720 MJPG failed: Device or resource busy
```

优先检查是否有 GUI、实时窗口、抓帧脚本同时运行。不要并行跑两个使用 `/dev/video73` 的命令。

## 模型类别与 ROI

- 当前缺陷模型只有 4 类：

```text
ZF-scratch
scratch
broken
pinbreak
```

- 当前模型不会检测整颗芯片，因为没有 `chip` 类。
- 当前画面直接全图送模型容易无框：芯片和缺陷在 `1280x720` 全幅中占比太小。
- 已验证：同一张 MaixCAM 实拍图，全图 `broken` 响应接近 0；裁出芯片 ROI 后，`raw` 分支可得到 `broken≈0.35`。
- 当前建议流程：

```text
全图 -> 自动 chip ROI -> ROI 裁剪 -> raw + light_gamma_clahe -> 缺陷检测 -> 框映射回原图
```

- 现有训练集可以用暗区域/边缘分割生成 `chip` 伪标签，用于训练 1 类芯片定位模型；正式训练前需要抽检修正。

## 预处理策略

- 不要把强白平衡作为默认唯一输入。实测强白平衡会把当前破损样本从 `broken` 推向 `ZF-scratch`。
- 当前默认两路更稳：

```text
raw                 保留破损类响应
light_gamma_clahe   辅助划痕/引脚/字符对比
```

- 预处理要和训练分布一致。若最终部署固定使用 `light_gamma_clahe`，后续训练/验证也应加入相同或相近的增强。
- 对字符污损、引脚破损、封装破损，理想上应分任务选择不同预处理；单个检测模型一次只能看一种输入表示，因此短期用两路推理合并结果。

## 框大小与精细定位

- 当前模型输出的是 bbox，不是 mask。框偏大不一定是画图问题，往往是模型和训练标签本身学到的是粗区域。
- 提高 `--conf` 可以减少低置信大框，但不能把 bbox 变成精确轮廓。
- 若需要精确缺角、断脚、污损轮廓，路线应是：

```text
chip ROI 定位
 -> 缺陷 bbox 粗检
 -> 按类别用轮廓/掩膜/分割模型精细化
```

- `broken` 可用芯片外轮廓 mask 和理想封装轮廓差分找缺角。
- `pinbreak` 可先定位引脚序列，再判断缺失、断裂、异常短脚。
- `scratch/字符污损` 更适合 ROI 内轻增强后用分割模型或局部纹理检测。

## Astra Pro Plus 注意点

- Astra RGB UVC 能作为普通相机输入；深度传感器当前 OpenNI2 打不开，缺少 Orbbec 私有驱动。
- 当前芯片检测暂不依赖深度；深度对微小表面缺陷帮助有限。
- Astra RGB 固定焦距/广角，对微小缺陷不如近景方案；应优先处理成像距离、目标比例和 ROI。
